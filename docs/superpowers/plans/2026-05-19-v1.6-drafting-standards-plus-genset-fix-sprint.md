## Reference table — JSON shape conventions (verified against live repo)

Every engine-lookupable JSON in `shared/standards/` follows this shape (extracted from `electrical/BS7671/diversity-factors.json`):

```jsonc
{
  "clause_ref": "<full citation, e.g., 'BS 1192:2007+A2:2016 §6 Layer Naming'>",
  "transcribed_at": "2026-05-19",
  "transcribed_by": "DraftsMan skills repo",
  "verification_status": "verified-against-source" | "draft-pending-verification" | "draft-from-secondary-needs-second-source-pair",
  "_title": "<engineer-facing title>",
  "_source": "<full source attribution with edition + year>",
  "_note": "<engineer-facing guidance, when standards diverge or interpretation is required>",
  "_units": "<units when applicable>",
  "_cross_refs": ["<other standard short codes this file references, one direction only>"],
  "<primary_key>": {
    "<lookup_key>": { <data fields> }
  },
  "engine_lookup": {  // OPTIONAL flat key/value form for runtime consumption
    "_purpose": "Flat key/value form for runtime engine consumption",
    "_reference": "<canonical source attribution>",
    "<flat_key>": { <flat data> }
  }
}
```

**Verification status enum (must use one of these exact values):**
- `verified-against-source` — author confirmed values against the purchased standard
- `draft-pending-verification` — author drafted from secondary source (e.g., AIA web pages, BSI summary); needs engineer review
- `draft-from-secondary-needs-second-source-pair` — single secondary source only; needs second-source check

**Cross-references convention:** one direction only. ISO 5457 references ISO 7200 (`_cross_refs: ["ISO7200"]`), but ISO 7200 does NOT add a reverse reference back to ISO 5457.

**Citation form in `clause_ref`:** Use full standard citation with year + amendment. Examples:
- `"BS 1192:2007+A2:2016 §6.3 Layer Naming Convention"`
- `"ISO 19650-2:2018 §5.1.2 Information Container Naming"`
- `"AIA CAD Layer Guidelines 2007 — Discipline-MajorGroup-MinorGroup-Modifier-Status"`

### Engineering correction (per IEC 60909-0:2016 §3.5.1)

**Calculation:**
- Genset: 80 kVA, 400 V TPN, salient-pole 4-pole synchronous machine
- In (full-load) = 80,000 / (√3 × 400) ≈ 115.5 A
- Subtransient reactance Xd" ≈ 0.12 pu (typical for 4-pole synchronous machine per IEC 60909-1:2002 Table A.1)
- Subtransient EMF E" ≈ 1.05 pu (per IEC 60909-0:2016 §3.5.1 equivalent voltage source)
- **Subtransient Ik" = In × E"/Xd" ≈ 115.5 × (1.05 / 0.12) ≈ 1.01 kA**
- Peak asymmetric ip ≈ κ × √2 × Ik" with κ ≈ 1.8 (low X/R ratio at genset terminals) → **ip ≈ 2.57 kA**
- Sustained Ik (5 s after fault, AVR field forcing settles) ≈ 0.4 kA (3-5 × In; this is what the previous "3-4× rated current" claim referenced — but it's sustained Ik, NOT subtransient Ik")

**Replacement template (use consistently across all 3 files):**

> Genset-mode subtransient Ik" ≈ 1.0 kA (IEC 60909-0:2016 §3.5.1 equivalent voltage source with Xd" = 0.12 pu, E" = 1.05 pu); peak asymmetric ip ≈ 2.5 kA (κ ≈ 1.8). Sustained Ik (after AVR field forcing, ~5 s post-fault) ≈ 0.4 kA (3-5× In). The earlier "3.5-4.5 kA" figure conflated sustained Ik with subtransient Ik".

### Step 8 — Write terminology.md

```markdown
# BS 1192 / ISO 19650 — Key Terminology

**Source:** BS 1192:2007+A2:2016 §3 (Definitions); aligned with ISO 19650-1:2018 §3 (Terms and definitions).

## Information Management

| Term | Definition |
|---|---|
| **WIP** | Work in Progress — pre-shared workspace for each individual task team |
| **Shared** | Reviewed and approved information shared between task teams |
| **Published** | Authorized for use by all project parties for the project's intended purpose |
| **Archive** | Documents superseded by newer versions; retained for record |
| **CDE** | Common Data Environment — the agreed source of information for the project, used to collect, manage and disseminate project information |

## Information Requirements

| Term | Definition |
|---|---|
| **EIR** | Exchange Information Requirements — pre-appointment requirements set by the appointing party |
| **AIR** | Asset Information Requirements — what's needed for the operational phase |
| **PIR** | Project Information Requirements — what's needed during the delivery phase |
| **OIR** | Organizational Information Requirements — strategic requirements of the appointing party |
| **PIM** | Project Information Model — the information model during the delivery phase |
| **AIM** | Asset Information Model — the information model during the operational phase |

## Suitability

See `status-codes.json` for the full S0-S7 enumeration.

## Cross-references

- `meta.json` — BS 1192 publication metadata
- `layer-naming.json` — BS 1192 §6 layer-naming convention (Role-Discipline-Number-Type)
- `file-naming.json` — BS 1192 §7 file-naming convention
- `revision-conventions.json` — BS 1192 §8 P/C-regime revision sequencing
- `../ISO19650/terminology.md` — ISO 19650 has a parallel terminology section (largely aligned)
```

### Step 12 — Write security-overview.md

```markdown
# ISO 19650-5 — Security-Minded Approach to Information Management

**Source:** ISO 19650-5:2020

## Why Part 5 exists

Building information shared through a CDE can be misused if it falls into the wrong hands. ISO 19650-5 establishes a **security-minded approach** that determines (a) whether security controls are needed for a given project and (b) what proportionate controls to apply.

## When Part 5 applies

A security-minded approach is required (not just recommended) when any of these trigger conditions apply:

1. **Critical national infrastructure** — energy, water, telecoms, transport, food, finance, defence, government, civil nuclear, emergency services
2. **Defence + security-related assets** — military, intelligence
3. **Public-facing buildings with high-occupancy or high-value contents** — stadiums, museums, transport hubs
4. **Information about people** — privacy considerations (data protection law)
5. **Commercially sensitive information** — competitive advantage, intellectual property

## The 5-step process

1. **Identify trigger** — Does any of the above apply?
2. **Sensitivity assessment** — What aspects are sensitive? What's the risk if disclosed?
3. **Define security strategy** — Cryptographic, access control, governance, technical controls
4. **Implement controls** — Apply ISO/IEC 27000 series technical controls + project-specific controls
5. **Audit + review** — Continuous review; controls adjust as project context changes

## What Part 5 is NOT

- NOT a cybersecurity standard — refers to **ISO/IEC 27000 series** for technical controls
- NOT a privacy/data-protection standard — refers to applicable law (GDPR in EU, etc.)
- NOT a security-classification scheme — uses project-specific or jurisdiction-specific classifications

## Cross-references

- `part5-security.json` — engine-lookupable assessment process + classification placeholders
- `cde-workflow.json` — CDE states subject to security controls
- `../BS1192/terminology.md` — BS 1192 terminology equivalents
```
