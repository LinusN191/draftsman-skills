## 1. Why this sprint

The earthing skill v1.0.0 shipped with TN-S enum support, per-circuit Zs verification, and 3 jurisdictional examples (UK TN-C-S, INT TT, US NEC). Two genuine gaps remain:

1. **No worked example exercises TN-S as the primary supply system.** The US example uses TN-S terminology but follows NEC bonding (different engineering content). UK TN-S supplies (separate-Earth from substation, Ze ≈ 0.80 Ω) demand a dedicated worked example.

2. **Zs computation is inline LLM math.** The new IEC 60364-5-52 Annex B cable impedance table (shipped at `33d1002`) gives per-CSA AC R+X values that the LLM can't carry in working memory. A deterministic `calc.zs_loop_impedance` tool is the right home for the computation.

Additionally, the user has flagged a forward-looking strategy decision: since the product will be consumed in Kenya first, INT examples should be Kenya-cued (KS 1700 + IEC 60364) where natural. This sprint kicks off that policy with the new earthing example.

## 3. Calc tool contract — `calc.zs_loop_impedance`

**File:** `shared/calculations/electrical/zs-loop-impedance.json`
**Executor mode:** `tool` (per WI3)
**Tool-call status:** pending (until DraftsMan runtime ships Python implementation)

### Why a tool, not inline

The LLM cannot carry full per-CSA AC R+X tables in working memory accurately. Inline computation in v1.0.0 used rough Ω/m estimates per CSA, accurate to ±15-25%. The deterministic tool reads `IEC60364/part5-52-cable-impedance.json` (just shipped) for IEC/BS/KE jurisdictions and `NFPA70/chapter9-tables.json` for US, applies temperature correction, returns precise Zs per circuit + Table 41.2 lookup verdict in one call.

### Input contract

```json
{
  "jurisdiction": "GB | EU | INT | KE | US",
  "supply": {
    "ze_declared_ohm": "float",
    "system_type": "TN-S | TN-C-S | TT",
    "voltage_phase_to_earth_v": "230 | 240 | 277",
    "frequency_hz": "50 | 60"
  },
  "circuits": [
    {
      "circuit_id": "string",
      "cable_material": "Cu | Al",
      "cable_insulation": "PVC | XLPE",
      "line_csa_mm2_or_awg": "string",
      "cpc_csa_mm2_or_awg": "string",
      "route_length_m": "float",
      "device_type": "MCB_B | MCB_C | MCB_D | BS_88_fuse | RCBO | GFCI",
      "device_rating_a": "int"
    }
  ],
  "operating_temperature_c": "int (default 70 PVC / 90 XLPE)"
}
```

### Output contract

```json
{
  "tool_version": "1.0.0",
  "computed_at": "ISO-8601",
  "sources": [
    "shared/standards/electrical/IEC60364/part5-52-cable-impedance.json",
    "shared/standards/electrical/BS7671/reg411-disconnection-times.json",
    "shared/standards/electrical/NFPA70/chapter9-tables.json"
  ],
  "results": [
    {
      "circuit_id": "string",
      "r1_per_km_ohm": "float (from impedance table)",
      "r2_per_km_ohm": "float (from impedance table)",
      "r1_plus_r2_ohm": "float ((r1+r2)/1000 × route_length_m)",
      "zs_calc_ohm": "float (Ze + R1+R2)",
      "zs_max_ohm": "float (from Table 41.2 / chapter9 ratio)",
      "zs_max_source": "string (e.g. BS7671 Table 41.2 type_B 32A)",
      "margin_pct": "float ((zs_max - zs_calc) / zs_max × 100)",
      "verdict": "pass | fail_needs_rcd | fail_oversize_cpc | pass_with_rcd",
      "rcd_required": "boolean"
    }
  ],
  "overall_compliant": "boolean"
}
```

### Jurisdiction branching

- **GB, KE, EU, INT** — uses `IEC60364/part5-52-cable-impedance.json` for cable R+X. Uses `BS7671/reg411-disconnection-times.json::table_41_2_MCB_Zs_max_ohms` for Zs_max lookup (KS 1700 Annex E reproduces BS Table 41.2 verbatim — same numbers).
- **US** — uses `NFPA70/chapter9-tables.json::Table_9_Alternating_Current_Resistance_and_Reactance_for_600V_Cables` for cable R+X. NEC ground-fault loop has no fixed `Zs_max` table — verdict derived from instantaneous trip current Ia at 5× rating with 600V phase-to-ground voltage.

### Adverse condition factor

For BS/KE jurisdictions, BS 7671 Section 411.3.2 implies a 0.8 factor on Zs_max already baked into Table 41.2 values (cable temperature rise at fault). Tool does NOT re-apply — the table values are already the design ceiling.

## 4. Kenya TN-S worked example

### Site brief

Nairobi industrial unit, Industrial Area (Enterprise Road), 1980s build. Light engineering workshop with mixed lighting + small power + 3-phase motor loads. Supplied from a KPLC 11 kV / 415 V substation 180 m away via armoured separate-Earth cable (TN-S — not PME).

### Standards stack

- **Primary:** KS 1700:2018 (Code of Practice for Electrical Wiring of Buildings)
- **Adopted by reference:** BS 7671:2018+A2:2022 (where KS 1700 defers)
- **Equipment grounding:** IEC 60364-5-54 (CPC sizing fallback)

KS 1700 Annex E reproduces BS 7671 Table 41.2 (Zs_max) verbatim, so the deterministic verification follows BS table values exactly.

### Supply data

| Field | Value |
|---|---|
| DNO | KPLC (Kenya Power & Lighting Company) |
| Wayleave reference | KPLC-NRB-IND-2143 (illustrative) |
| Connection year | 1987 |
| System type | TN-S (separate-Earth cable from substation transformer star point) |
| Voltage | 415V/240V 3-phase 4-wire 50 Hz |
| Declared Ze | 0.80 Ω (measured at intake position; degraded from original 0.65 Ω by sheath corrosion) |
| Main switch | 100A TPN with shrouded busbars |
| PFC declared | 9.8 kA |

### Circuits (8 total)

| ID | Description | Phase | MCB | Cable | Length | CPC |
|---|---|---|---|---|---|---|
| C01 | Lighting workshop (LED highbays) | L1 | 10A Type B | 1.5mm² T+E | 35m | 1.0mm² |
| C02 | Lighting office mezzanine | L2 | 6A Type B | 1.5mm² T+E | 22m | 1.0mm² |
| C03 | Socket ring main workshop | L3 | 32A Type B | 2.5mm² T+E | 28m loop | 1.5mm² |
| C04 | Socket radials office | L1 | 20A Type B | 2.5mm² T+E | 18m | 2.5mm² |
| C05 | Lathe 3-ph (5.5kW motor) | L1+L2+L3 | 20A Type C | 2.5mm² SY 4-core | 15m | 2.5mm² |
| C06 | Pillar drill 3-ph (2.2kW) | L1+L2+L3 | 16A Type C | 1.5mm² SY 4-core | 12m | 1.5mm² |
| C07 | Compressor 3-ph (7.5kW) | L1+L2+L3 | 25A Type D | 4mm² SY 4-core | 25m | 4mm² |
| C08 | Submain to gate house | L1+N+CPC | 40A Type B | 10mm² SWA | 60m | 6mm² SWA armour |

### Engineering decisions exercised

1. **TN-S vs TN-C-S split** — Ze 0.80 Ω (TN-S separate-Earth) vs 0.35 Ω (TN-C-S PME) shifts compliance margin on C07 (Type D 25A) from "comfortable pass" to "tight pass / requires RCD". Demonstrates why TN-S is harder to verify than TN-C-S.
2. **Type D motor MCBs** — Ia = 20×In demands lower Zs_max (Zs_max = 230/(20×25) = 0.46 Ω for 25A) → C07 forces 4mm² CPC instead of 2.5mm² that an inattentive engineer might select.
3. **Submain length impact** — C08 60m run with 6mm² CPC: R1+R2 ≈ 0.6 Ω → Zs ≈ 1.4 Ω vs Type B 40A Zs_max = 1.15 Ω → **fails Zs without RCD**, requires 30mA RCBO additional protection.
4. **KE-specific RCD policy** — KS 1700 §411.3.3 mandates RCD on all socket circuits ≤32A regardless of Zs compliance (mirrors BS 7671 AMD 2). Rationale documents this explicitly with code clause.
5. **Tool deferral pattern** — `tool_call_pending_for_zs: true` flag set; computed Zs values are LLM-inline estimates with explicit rationale disclaimer.

## 5. IR schema additions

**File:** `electrical/earthing/schemas/earthing-ir.schema.json`

Two new optional fields at IR root. No breaking changes — v1.0.0 outputs (which lack these) remain valid.

```json
{
  "tool_call_pending_for_zs": {
    "type": "boolean",
    "description": "WI3 deferral flag. True until calc.zs_loop_impedance runtime tool ships. When true, circuits[].zs_ohm and zs_max_ohm are LLM-computed estimates with disclaimer in rationale.flags."
  },
  "zs_calc_tool_input": {
    "type": "object",
    "description": "The payload that would be sent to calc.zs_loop_impedance once runtime ships. Captured at generation time so the runtime can replay deterministically without re-prompting the LLM.",
    "required": ["jurisdiction", "supply", "circuits", "operating_temperature_c"],
    "properties": { /* matches §3 input contract */ }
  }
}
```

**Per-circuit additions:** none. Existing `circuits[].zs_ohm`, `zs_max_ohm`, `zs_compliance`, `rcd_required` fields stay. Once runtime tool ships, runtime can overwrite them with deterministic values without IR shape changes.

**`rationale.flags` convention** (existing field, new well-known value):
- When `tool_call_pending_for_zs: true`, `rationale.flags` MUST contain a string matching `^TOOL-CALL-PENDING:calc\.zs_loop_impedance`

## 7. Generator prompt changes

**File:** `electrical/earthing/prompts/generator.md`

### Insert Step 9.5

> **Step 9.5 — Build the calc.zs_loop_impedance request payload (WI3 deferred)**
>
> Assemble `zs_calc_tool_input` object per §3 input contract. Populate from upstream supply data + circuit list. This is the snapshot the runtime tool will consume once shipped.
>
> Fields to populate:
> - `jurisdiction` (from inputs)
> - `supply.{ze_declared_ohm, system_type, voltage_phase_to_earth_v, frequency_hz}` (from supply data)
> - `circuits[].{circuit_id, cable_material, cable_insulation, line_csa_mm2_or_awg, cpc_csa_mm2_or_awg, route_length_m, device_type, device_rating_a}` (from each circuit)
> - `operating_temperature_c` (70 for PVC, 90 for XLPE; default 70 if mixed)

### Reshape Step 10

> **Step 10 — Compute Zs and verdict inline, with WI3 disclaimer**
>
> Since `calc.zs_loop_impedance` is not yet runtime-shipped (WI3 deferral), compute Zs per circuit inline using rough R1+R2 estimates from the cable CSA + length. Apply BS 7671 Table 41.2 lookup for Zs_max (or NEC equivalent ratio for US).
>
> **Required actions:**
> - Set `tool_call_pending_for_zs: true` at IR root
> - Populate each `circuits[].{zs_ohm, zs_max_ohm, zs_compliance, rcd_required}` with the inline estimate
> - Append to `rationale.flags`: `"TOOL-CALL-PENDING:calc.zs_loop_impedance — Zs values are LLM-estimates; deterministic refinement deferred per WI3."`
>
> Once runtime tool ships, this step becomes: invoke the tool with `zs_calc_tool_input`, populate `circuits[]` from tool response, set `tool_call_pending_for_zs: false`, remove the TOOL-CALL-PENDING flag.

## 8. Validator prompt changes

**File:** `electrical/earthing/prompts/validator.md`

Add new INV check (matching the 8-INV pattern from arc-flash validator):

> **INV-9 — Tool deferral shape**
>
> If `tool_call_pending_for_zs: true`:
>   - `zs_calc_tool_input` MUST be present and well-formed per §3 input contract
>   - `rationale.flags` MUST contain a string matching `^TOOL-CALL-PENDING:calc\.zs_loop_impedance`
>

>   - The TOOL-CALL-PENDING string MUST NOT be present in `rationale.flags`
>   - `circuits[].zs_ohm` values must reflect tool output (not LLM estimates)

## 9. New validator YAML

**File:** `electrical/earthing/validation/tool-deferral-shape.yaml`

Two deterministic checks:

```yaml
checks:
  - id: tool_deferral_pair_consistent
    severity: critical
    description: "When tool_call_pending_for_zs is true, zs_calc_tool_input + TOOL-CALL-PENDING flag must both be present"
    rule: |
      if ir.tool_call_pending_for_zs == true then
        ir.zs_calc_tool_input must exist AND
        ir.rationale.flags must include string matching ^TOOL-CALL-PENDING:calc\.zs_loop_impedance

  - id: tool_deferral_no_orphan_flag
    severity: critical
    description: "TOOL-CALL-PENDING:calc.zs_loop_impedance flag must not appear without tool_call_pending_for_zs being true"
    rule: |
      if ir.rationale.flags includes string matching ^TOOL-CALL-PENDING:calc\.zs_loop_impedance then
        ir.tool_call_pending_for_zs == true
```
