## 1. Why this sprint

The earthing skill (and several others) declares `consumes_intents: [...]` in its manifest, but no worked example actually demonstrates the consumption flow. Circuits in the earthing examples are hand-authored from scratch in `input.json` rather than derived from an upstream `db-layout` intent. This means the WI4 cross-drawing intent contract — defined in ARCHITECTURE.md and the retired upstream-contributions spec — is currently documented but unused.

This sprint operationalizes the pattern. After v1.3, all 4 earthing examples exercise db-layout intent consumption end-to-end, providing canonical demonstrations that any future consumer-skill sprint can reference.

Additionally, this sprint fills a gap discovered during the v1.1 earthing retrofit: db-layout's 3 worked examples have `input.json` + `output.json` but no `intent-out.json` file. Without that file, no downstream skill can actually consume the intent.

## 3. Cross-skill data flow (4 stages)

### Stage 1 — db-layout authors the upstream

```
electrical/db-layout/examples/<scenario>/
├── input.json          (engineer-entered scenario brief)
├── output.json         (full db-layout IR)
├── intent-out.json     (NEW: slim downstream-consumable subset)
└── reasoning.md
```

`intent-out.json` carries: `circuits[]` with `{ circuit_id, breaker_rating_a, breaker_curve, voltage_class, route_length_m_approx, cable_csa_mm2 }` per row. This is the canonical contract published by db-layout.

### Stage 2 — earthing input declares the upstream

`electrical/earthing/examples/<scenario>/input.json` adds:

```json
"consumed_intent_path": "electrical/db-layout/examples/<paired>/intent-out.json"
```

Relative path from repo root. Engineer declares lineage explicitly.

### Stage 3 — earthing output derives circuits from the upstream

The earthing IR's `circuits[]` are aligned to the upstream:
- `circuit_id` MUST be a subset of upstream `intent-out.circuits[].circuit_id` (no new IDs introduced; filtering allowed for ELV/comms)
- `breaker_rating_a`, `breaker_curve`, `voltage_class`, `route_length_m` come **from the upstream**
- Earthing-specific fields added: `cpc_csa_mm2_or_awg`, `cpc_sizing_method`, `cpc_sizing_clause`, `zs_ohm`, `zs_max_ohm`, `zs_compliance`, `rcd_required`, `rcd_type`, `rcd_sensitivity_ma`

### Stage 4 — earthing output records provenance

```json
"meta": {
  "consumed_intents": [
    {
      "intent_type": "db-layout",
      "intent_version": "1.0.0",
      "produced_by": "electrical/db-layout/v1.1.0"
    }
  ]
}
```

**Convention note:** `intent_version` is the intent schema/contract version (currently 1.0.0 — the intent shape has not changed). `produced_by` carries the producing skill version. The two are decoupled — minor skill version bumps that don't change the intent contract do NOT bump intent_version. Same pattern as `earthing/intent-out.json` which carries `intent_version: 1.0.0` + `produced_by_skill_version: earthing/1.1.0`.

### Runtime resolution status

The runtime that automatically resolves `consumed_intent_path` + injects the upstream intent payload into the LLM's context is part of the DraftsMan runtime (not this skills repo). For now:
- The path is documented in input.json
- The LLM (when generating a new design) sees the path + reads the file at the path (or the runtime injects it)
- For the worked examples in this sprint, the circuits[] are pre-authored to match what resolved consumption would produce

When the runtime ships, no IR change needed — runtime just automates the resolution.

### E. Earthing generator prompt update

Add new step before existing Step 1:

```markdown
## Step 0.5 — Resolve upstream intents (if consumed_intent_path declared)

If `input.consumed_intent_path` is present:
1. Read the referenced intent-out.json file
2. Adopt the upstream `circuits[]` as the basis for your `circuits[]` (use `circuit_id`, `breaker_rating_a`, `breaker_curve`, `voltage_class`, `route_length_m` from the upstream verbatim — do NOT re-author these)
3. Populate `output.meta.consumed_intents[]` with `{intent_type: "db-layout", intent_version: <from upstream>, produced_by: "electrical/db-layout/v1.1.0"}`
4. Filter the upstream circuits if any are out-of-scope for earthing (e.g., ELV control circuits) — document the filtering in `reasoning.flags`
5. Extend each retained circuit with earthing-specific fields: CPC sizing, Zs verification, RCD specification (Steps 6-10 below)

If `consumed_intent_path` is NOT present:
- Author circuits from the input.json brief as before (v1.0-1.2 behaviour)
- Leave `meta.consumed_intents` empty
```

## 5. Critical alignment constraint

For all 4 earthing examples post-refresh:

**`earthing.circuits[].circuit_id` MUST be a subset of `db-layout.intent-out.circuits[].circuit_id`** (verified by eval-09 assertion 3-6).

The earthing skill MAY filter circuits but MUST NOT introduce new circuit IDs that weren't in the upstream intent. Where filtering occurs, the filtered circuit IDs are documented in `compliance_summary.assumptions[]` as `"Circuit <ID> filtered from upstream db-layout intent — out of scope for earthing (e.g., ELV control)"`.
