## 1. Why this sprint

SLD v1.3 (shipped 2026-05-18) consumes db-layout intents only — one per board in the cascade. This demonstrated the WI4 multi-board pattern. v1.4 generalizes the pattern from single-skill to multi-skill: the SLD aggregates data from THREE upstream skill domains (db-layout for per-board detail, earthing for system-wide earthing context, fault-level for deterministic PFC).

Critical engineering benefit: SLD's `system_metrics.peak_pfc_ka` field — currently an LLM inline estimate with WI3 deferral flag — gets refined to a deterministic value sourced from the fault-level skill's calc.iec60909_cascade output. This is the first concrete example of how multi-skill provenance feeds engineering precision, validating the cross-drawing intent architecture beyond pure documentation.

Additionally, the fault-level skill has the same intent-out backfill gap that db-layout had before earthing v1.1 — 3 existing examples (intl-commercial-with-genset, uk-domestic-single-source, us-industrial-with-motors) lack intent-out.json files. SLD v1.4 sprint includes the fault-level v1.1 backfill phase.

## 3. Architecture decisions

1. **Multi-skill consumption: each SLD example consumes 3 intent types.** N db-layout intents (one per board) + 1 earthing intent (system-wide) + 1 fault-level intent (system-wide). `meta.consumed_intents[]` length is N+2 entries (N db-layout + 1 earthing + 1 fault-level). For UK SLD with 4 boards: 4 + 1 + 1 = 6 entries.

2. **input.json gains two new top-level fields.** `earthing_intent_path` + `fault_level_intent_path` — relative paths from repo root pointing at the earthing/fault-level skill's intent-out.json for this project.

3. **system_metrics.peak_pfc_ka refined from fault-level intent.** In v1.3 this was an LLM inline estimate. In v1.4, the value comes from the fault-level intent's `ik3_max_ka_at_intake` (or equivalent peak PFC at MSB). Other system_metrics fields (imax_total, SPD assessment, life-safety isolation) stay LLM-estimated with `tool_call_pending_for_system_metrics: true` until calc.sld_system_metrics ships.

4. **Cross-file consistency rules (validator INV-11):**
   - `meta.consumed_intents[]` length == `distribution_hierarchy.length + 2`
   - Exactly 1 entry with `intent_type == "earthing"`
   - Exactly 1 entry with `intent_type == "fault-level"`
   - `earthing_intent_path` + `fault_level_intent_path` resolve to existing files
   - SLD `supply_origin.system_type` matches earthing intent's `earthing_system.system_type`
   - SLD `system_metrics.peak_pfc_ka` matches fault-level intent's peak PFC (±0.5 kA tolerance)

5. **Backward-compatible.** SLD v1.3 examples (with only db-layout consumption) remain valid; INV-11 only applies when `earthing_intent_path` AND `fault_level_intent_path` are declared in input. SLD v1.4 examples carry both paths; SLD v1.3 examples leave them absent.

6. **Cross-domain pairing strategy: build NEW commercial-aligned upstream examples for tension cases.** Per user direction 2026-05-18 (Option 2 from brainstorming):
   - UK SLD (commercial) ↔ NEW uk-commercial-3storey earthing + NEW uk-commercial-3storey fault-level
   - KE SLD (industrial) ↔ existing ke-nairobi-industrial-tn-s earthing + NEW ke-nairobi-industrial fault-level
   - INT SLD (commercial MSB) ↔ existing intl-rural-tt earthing (re-anchored commercial v1.3) + existing intl-commercial-with-genset fault-level (partial alignment, genset filtering documented)
   - US SLD (retail strip mall) ↔ existing us-commercial-nec earthing (commercial NEC — close enough) + NEW us-strip-mall-retail fault-level

7. **fault-level v1.1 backfills intent-out.json for the 3 existing examples** even though only 1 is consumed by SLD v1.4 (intl-commercial-with-genset). The other 2 (uk-domestic-single-source, us-industrial-with-motors) get backfilled for completeness — same pattern as db-layout v1.1 backfill before earthing v1.1.

## 8. Generator prompt update — Step 0.5 extended

The SLD v1.3 generator prompt's Step 0.5 ("Read upstream intent paths") gets extended to handle 3 intent types. Replacement structure:

**A. Read each db-layout intent (one per board).**
- For each `db_layout_intent_paths[i]`, load intent-out.json; verify `intent_type == "db-layout"`; capture board_id, supply_origin.board_id, ocpd/cable per circuit, total_load_kva.
- Pre-existing v1.3 behavior — unchanged.

**B. Read the earthing intent (system-wide).**
- Load `earthing_intent_path` intent-out.json; verify `intent_type == "earthing"`.
- Capture `earthing_system.system_type` → cross-check against SLD `supply_origin.system_type` (must match exactly).
- Capture `earthing_system.supply_bond_type` → seed SLD `supply_origin.supply_bond_type`.
- Capture `earthing_system.ze_ohm` (if present) → seed SLD `supply_origin.ze_ohm`.
- Capture SPD policy inputs (if present) → inform SLD `compliance_summary.spd_assessment.recommendation`.

**C. Read the fault-level intent (system-wide).**
- Load `fault_level_intent_path` intent-out.json; verify `intent_type == "fault-level"`.
- Capture `ik3_max_ka_at_intake` (or equivalent top-of-cascade peak PFC field) → set `system_metrics.peak_pfc_ka` (replaces v1.3 LLM estimate).
- Capture per-board cascade PFC values (if present) → inform per-board breaking-capacity sizing.
- Capture method chain used (IEC 60909 vs. table) → inform reasoning narrative.

**D. Populate `meta.consumed_intents[]` in order:** N db-layout entries (one per board, in parent-first cascade order) → 1 earthing entry → 1 fault-level entry. Total length = N+2.

**E. Generate cross-skill assumptions.** For each cross-skill input, add a `compliance_summary.assumptions[]` entry naming the source skill + intent path + the field consumed. For INT (genset filtering case): add an explicit assumption stating SLD models utility-source PFC only and genset contribution is documented upstream.

## 9. Validator update — new INV-11

INV-11 fires only when both `earthing_intent_path` AND `fault_level_intent_path` are declared in input (backward-compatible with v1.3 examples). Checks (per design § 3 decision 4):

1. `meta.consumed_intents.length == distribution_hierarchy.length + 2`
2. Exactly 1 entry with `intent_type == "earthing"`
3. Exactly 1 entry with `intent_type == "fault-level"`
4. `earthing_intent_path` resolves to an existing file (relative to repo root)
5. `fault_level_intent_path` resolves to an existing file
6. SLD `supply_origin.system_type` equals earthing intent's `earthing_system.system_type` (string equality)
7. SLD `system_metrics.peak_pfc_ka` within ±0.5 kA of fault-level intent's peak PFC field

Failure of checks 1–5 is a hard fail (compliance flag `INV-11:multi-skill-shape-fail`). Failures of 6–7 emit a warning flag (`INV-11:cross-skill-mismatch`) — not blocking, since engineer override may be valid.
